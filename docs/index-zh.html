Claude Code 使用者面臨的問題

CCB 透過智能記憶和能力路由解決常見挫折

問題：上下文遺失

Claude 會忘記之前的對話。你必須在每個 session 中重複說明專案架構、編碼標準和決策。

解決方案：專案記憶

知識圖譜永久記住你的架構決策、編碼模式和過去的解決方案，跨越所有 session。

實際範例：「為什麼我們選擇 PostgreSQL？」→ CCB 回憶完整決策，包含考慮過的替代方案和權衡分析

問題：泛化的回應

無論你是在除錯、審查程式碼還是設計 UI，Claude 的回應都一樣泛化。缺乏專業深度。

解決方案：智能任務路由

依能力路由提供專家級回應。程式碼審查專注安全，除錯提供系統性分析，UI/UX 設計強調可用性與一致性。

實際範例：「審查此程式碼」→ 自動路由到 code review 能力 → 獲得安全檢查清單 + OWASP 最佳實踐 + 具體建議

問題：昂貴的 Token 成本

所有任務都使用昂貴的模型，即使是簡單工作也浪費 token 和預算。

解決方案：智能模型選擇

依複雜度路由任務：Haiku（簡單、便宜）→ Sonnet（平衡）→ Opus（複雜、高品質）。節省約 40% token 成本。

實際運作：TaskAnalyzer 分析任務複雜度 → 自動選擇最適合的 Claude 模型 → 簡單代碼用 Haiku，複雜架構用 Opus

問題：重複的上下文說明

每次對話都要重新解釋專案結構和決策，浪費時間、打斷工作流程。

解決方案：專案記憶與自動整理

透過 hooks 在關鍵階段自動整理專案記憶，保留決策與上下文，不需重複說明。

實際運作：HookIntegration 捕捉進度 → Project Memory 生成 → buddy remember 查詢相關記憶

專為隨您的專案演化而設計

CCB 不再將每個提示視為全新的開始，而是保持對您的程式碼庫、架構決策和工作流程模式的持久理解。

隨著您的使用，Plugin 會逐漸與您的專案結構和工作偏好保持一致—減少重複、提高一致性，讓 AI 協助感覺更不像是一個工具，而更像是一個長期的合作夥伴。

運作原理：行為演化

CCB 不會重訓模型。相反地，它使用 Plugin 內部的 持久化系統層：

專案記憶累積

架構決策、慣例和已解決的討論都被儲存為結構化的上下文。

模式強化

重複的批准、重構和偏好的解決方案逐漸塑造未來提示詞的建構方式。

任務類型區分

審查、除錯、重構和設計任務使用不同的內部提示策略處理。

上下文重用

選擇性地注入相關記憶，避免提示詞過其臃腫，同時保持連續性。

快速開始

只需 2 分鐘即可開始使用

前置需求

Node.js 20+ 和 Claude Code 已安裝

# 檢查 Node.js 版本
node --version  # 應為 v20 或更高

複製與安裝

git clone https://github.com/PCIRCLE-AI/claude-code-buddy.git

# 執行互動式安裝程式（處理所有事項）

重新啟動 Claude Code 並嘗試這些命令：

"分析我的程式碼庫架構"
"為 auth.ts 生成測試"
"審查此代碼的安全性"
"優化此資料庫查詢"

如何使用 CCB

安裝完成後，CCB 透過兩種方式運作

自動模式

CCB 在背景自動工作。像平常一樣使用 Claude Code，CCB 會根據你的任務自動路由到合適的能力。

範例：當你請 Claude「審查這段程式碼」，CCB 自動路由到 code review 能力提供安全分析。

明確指令模式

使用 buddy 指令直接與 CCB 互動，獲取記憶管理、工作流程引導等功能。

範例：buddy remember 我們為什麼選擇 PostgreSQL？

Buddy 指令

buddy do [任務]

智能執行任務並路由到合適的能力

buddy remember [問題]

從專案記憶中召回相關資訊

buddy help

顯示所有可用指令的說明

記憶工具

buddy-remember

buddy-help
